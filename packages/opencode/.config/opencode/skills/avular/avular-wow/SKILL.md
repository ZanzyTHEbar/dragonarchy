# Avular Way Of Working

Use this skill as the entry point for Avular engineering work.

## Core Rule

Apply the minimum sufficient process for the size, risk, and cross-functional impact of the work.

Do not add process just because it exists.

## Routing

1. If the task is about solution design, architecture trade-offs, or review readiness, use `avular-design-docs`.
2. If the task is about where a document, ticket, formal deliverable, or shared note belongs, use `avular-doc-routing`.
3. If the task is about lifecycle stage readiness, stage-gate expectations, or evidence needed for progression, use `avular-lifecycle-gates`.
4. If the task is about product tiers, naming, compatibility language, or version semantics, use `avular-product-versioning`.

## Guardrails

- Prefer decisions, risks, interfaces, and verification over ceremony.
- Keep design output proportional to the change.
- Escalate only when the work crosses component boundaries, stakeholder boundaries, or lifecycle boundaries.
- If a topic does not affect software engineering decisions, do not encode it into the workflow.

## Output

- Which Avular skill applies
- Why it applies
- The smallest useful next artifact or decision
