# Plan: Generic Containerized OpenFortiVPN

## Goal

Distill the current dotfiles `openfortivpn` implementation into a **standalone, generic, containerized variant** that can be deployed on any Linux machine (systemd or not) via Docker or Podman. The containerized version must achieve **full feature parity** with the current implementation while improving portability, maintainability, and UX.

---

## Scope

| Feature | Current (repo) | Containerized Target |
|---|---|---|
| VPN engine | `openfortivpn` + `pppd` | Same, inside container |
| SAML login | `--saml-login=8020`, browser redirect | Same, port exposed on host |
| Split DNS | `systemd-resolved` + `resolvectl` | Portable DNS forwarder (`dnsmasq`) |
| Service lifecycle | `systemd` units (`openfortivpn.service`, `openfortivpn-cleanup.service`) | Container signal handling + health checks |
| Control interface | 566-line Waybar bash script | HTTP API + `vpnctl` CLI + optional Web UI |
| Config management | Ansible role seeds `/etc/openfortivpn/config` | Volume-mounted config + env vars + API |
| Notifications | `notify-send` (desktop-specific) | API events + optional Web UI toast |
| Cleanup | `pkill` chain in systemd oneshot | Graceful SIGINT handler + automatic DNS restore |

## Architecture

### High-Level Design

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              Host (Linux)                                │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────────────────────────┐│
│  │   Browser   │   │  vpnctl CLI │   │        Web Browser (opt)        ││
│  │  (SAML)     │   │             │   │                                 ││
│  └──────┬──────┘   └──────┬──────┘   └─────────────────────────────────┘│
│         │                 │                                              │
│         │                 └──────────────────┐                           │
│         │                                    │                           │
│         ▼                                    ▼                           │
│  ┌─────────────────────────────────────────────────────────────────────┐│
│  │                     OpenFortiVPN Container                         ││
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────────┐ ││
│  │  │ openfortivpn│  │  dnsmasq    │  │    Control Plane (Go)       │ ││
│  │  │   + pppd    │  │  (DNS fwd)  │  │  - HTTP API (Unix+TCP)      │ ││
│  │  │             │  │             │  │  - Config manager           │ ││
│  │  │ creates     │  │ listens on  │  │  - Process supervisor       │ ││
│  │  │ ppp0 on host│  │ 127.0.0.1:53│  │  - Signal handler           │ ││
│  │  └──────┬──────┘  └──────┬──────┘  └──────────────┬────────────────┘ ││
│  │         │                │                        │                  ││
│  │         └────────────────┴────────────────────────┘                  ││
│  │                      entrypoint.sh (PID 1)                           ││
│  └─────────────────────────────────────────────────────────────────────┘│
│         │                                                              │
│         │ --network host  --privileged  --pid=host (optional)          │
│         ▼                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐│
│  │  Host Network Stack                                                 ││
│  │  - ppp0 interface created by container's pppd                       ││
│  │  - /etc/resolv.conf managed by container (bind mount)               ││
│  │  - Routes set by openfortivpn/pppd (host-visible)                   ││
│  └─────────────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────────────┘
```

### Key Constraints

1. **PPP interfaces are kernel-level**: `pppd` creates `ppp0` in the network namespace where it runs. For the host to see the interface and route traffic through it, the container **must** use `--network host` (or `--pid=host` with special setup). There is no clean "veth" equivalent for PPP.
2. **Privilege requirement**: `openfortivpn` and `pppd` require `CAP_NET_ADMIN` and access to `/dev/ppp`. The container must run privileged or with explicit capabilities.
3. **macOS/Windows limitation**: `--network host` on Docker Desktop (macOS/Windows) does not expose the container's network interfaces to the macOS/Windows host. **Full-network VPN mode is Linux-only.** This must be documented clearly.

## Component Breakdown

### 1. Container Image (`Containerfile`)

**Base**: `alpine:3.19` or `debian:12-slim`
- Alpine is smaller but `openfortivpn` may need compilation.
- Debian has `openfortivpn` in repos (testing/backports) and `pppd` is well-supported.
- **Decision**: Start with `debian:12-slim` for reliability; optimize to Alpine later if needed.

**Installed packages**:
- `openfortivpn`
- `ppp`
- `dnsmasq`
- `iproute2`
- `curl`, `jq`
- `tini` (init system for proper signal handling)

**Control Plane**:
- A statically-linked Go binary (~5 MB) serving the HTTP API.
- Go is chosen because it compiles to a single static binary with no runtime dependencies, making the container clean.

**Entrypoint**: `tini -- /usr/local/bin/entrypoint.sh`
- `entrypoint.sh` is PID 1 (via `tini`) and:
  1. Validates environment/config.
  2. Starts `dnsmasq` in the background.
  3. Starts the control plane (Go binary) in the background.
  4. Waits for API `/connect` or auto-connect.
  5. Traps SIGINT/SIGTERM to trigger graceful shutdown.

### 2. DNS Strategy (Portable Split DNS)

**Current problem**: `avular-vpn-dns` requires `systemd-resolved` and `resolvectl`. Many distros (Alpine, Void, Artix) do not use systemd.

**Containerized solution**:
1. Run `dnsmasq` inside the container bound to `127.0.0.1:53`.
2. Mount the host's `/etc/resolv.conf` into the container (e.g., `/host/etc/resolv.conf`).
3. On VPN connect:
   - Back up the host's `/etc/resolv.conf`.
   - Write `nameserver 127.0.0.1` to the host's `/etc/resolv.conf`.
   - Configure `dnsmasq` with:
     - `server=/avular.dev/10.10.100.50`
     - `server=/avular.dev/10.10.100.11`
     - `server=/*/<original-primary-dns>` (fallback)
4. On VPN disconnect:
   - Restore the backed-up `/etc/resolv.conf`.
   - Clear `dnsmasq` VPN-specific config.

**Edge case handling**:
- If `/etc/resolv.conf` is a symlink (common on systemd hosts), resolve the real path and backup/restore that.
- If `resolvectl` is available on the host, prefer it as the primary path (cleaner), with `dnsmasq` as the portable fallback.
- Provide an env var `DNS_METHOD=auto|resolved|dnsmasq|none` to force a method.

### 3. Control Plane (Go HTTP API)

**Endpoints**:

```yaml
GET  /health          # Container health (200 = healthy)
GET  /status          # { state: "connected"|"connecting"|"disconnected", iface: "ppp0", ip: "10.x.x.x", uptime_seconds: 123 }
POST /connect         # Starts openfortivpn. Returns: { saml_url: "https://...", saml_port: 8020 }
POST /disconnect      # Stops openfortivpn gracefully. Returns: { state: "disconnected" }
GET  /config          # Returns current openfortivpn config as text
POST /config          # Updates config (body = raw config text). Requires restart.
GET  /logs            # Streams openfortivpn stdout/stderr (SSE or chunked)
GET  /saml/status     # Returns { ready: true|false } — whether SAML listener is up
```

**State machine**:
```
disconnected -> connect() -> connecting -> saml_ready -> connected
connected -> disconnect() -> disconnected
connecting -> timeout/error -> disconnected
```

**Implementation notes**:
- The Go binary supervises the `openfortivpn` process directly (using `os/exec`).
- It captures stdout/stderr and exposes them via `/logs`.
- It monitors `ppp0` interface existence via `netlink` or `ip link` polling.
- It exposes a Unix socket at `/run/openfortivpn/api.sock` for local CLI access.
- Optionally exposes TCP `127.0.0.1:8080` for Web UI access.

### 4. CLI Client (`vpnctl`)

A single-file shell script (or a small Go binary) that talks to the container API.

```bash
vpnctl status          # GET /status
vpnctl connect         # POST /connect; opens browser if URL returned
vpnctl disconnect      # POST /disconnect
vpnctl logs            # GET /logs
vpnctl config          # GET /config
vpnctl config --edit   # POST /config (launches $EDITOR)
vpnctl health          # GET /health
```

**Discovery**:
- Checks `OPENFORTIVPN_API_SOCKET` env var (default: `/run/openfortivpn/api.sock`).
- Falls back to `OPENFORTIVPN_API_URL` (default: `http://127.0.0.1:8080`).

### 5. Web UI (Optional)

A single-page HTML/JS app served by the control plane at `GET /ui/`.

**Features**:
- Status indicator (connected/disconnected/connecting)
- Connect / Disconnect buttons
- SAML login link (auto-opens on connect)
- Log viewer (tailing `/logs`)
- Config editor (textarea + save)

**Tech**: Pure HTML + vanilla JS. No build step. Embedded in the Go binary via `embed`.

### 6. SAML Login Flow

1. User calls `vpnctl connect` (or clicks Web UI "Connect").
2. Control plane starts `openfortivpn --saml-login=8020`.
3. API returns `{ saml_url: "https://vpn.avular.com:443/remote/saml/start?redirect=1" }`.
4. `vpnctl` (or Web UI) opens the URL in the default browser (`xdg-open` / `open`).
5. Browser completes SAML login and redirects to `http://127.0.0.1:8020`.
6. `openfortivpn` receives the token and establishes the tunnel.
7. Control plane detects `ppp0` up and applies DNS.

**Note**: Port `8020` is on the host because of `--network host`. No port mapping is needed.

## Host Integration Model

### Docker Compose (`docker-compose.yml`)

```yaml
services:
  openfortivpn:
    image: ghcr.io/daofficialwizard/openfortivpn:latest
    container_name: openfortivpn
    network_mode: host
    privileged: true
    pid: host
    cap_add:
      - NET_ADMIN
    devices:
      - /dev/ppp:/dev/ppp
    volumes:
      - ./config/openfortivpn.conf:/etc/openfortivpn/config:ro
      - /etc/resolv.conf:/host/etc/resolv.conf
      - openfortivpn-run:/run/openfortivpn
    environment:
      - VPN_HOST=vpn.avular.com
      - VPN_PORT=443
      - VPN_DOMAIN=avular.dev
      - VPN_DNS_SERVERS=10.10.100.50,10.10.100.11
      - SAML_PORT=8020
      - DNS_METHOD=auto
      - AUTO_CONNECT=false
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://127.0.0.1:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s

volumes:
  openfortivpn-run:
```

### Podman Quadlet (`openfortivpn.container`)

```ini
[Container]
Image=ghcr.io/daofficialwizard/openfortivpn:latest
Network=host
Privileged=true
PID=host
AddDevice=/dev/ppp
Volume=./config/openfortivpn.conf:/etc/openfortivpn/config:ro
Volume=/etc/resolv.conf:/host/etc/resolv.conf
Volume=openfortivpn-run:/run/openfortivpn
Environment=VPN_HOST=vpn.avular.com
Environment=VPN_PORT=443
Environment=DNS_METHOD=auto
HealthCmd=curl -f http://127.0.0.1:8080/health
HealthInterval=30s

[Service]
Restart=unless-stopped

[Install]
WantedBy=default.target
```

### Systemd Service (for hosts that prefer systemd over compose/quadlet)

A generated `openfortivpn.service` that runs the container via `podman run` or `docker run`. This is for users who want systemd integration without Quadlet.

## Configuration Model

### Environment Variables

| Variable | Default | Description |
|---|---|---|
| `VPN_HOST` | (required) | FortiGate hostname |
| `VPN_PORT` | `443` | FortiGate port |
| `VPN_DOMAIN` | `avular.dev` | Domain for split DNS |
| `VPN_DNS_SERVERS` | `10.10.100.50,10.10.100.11` | Comma-separated DNS servers |
| `SAML_PORT` | `8020` | Local port for SAML listener |
| `DNS_METHOD` | `auto` | `auto`, `resolved`, `dnsmasq`, `none` |
| `AUTO_CONNECT` | `false` | Start VPN immediately on container boot |
| `CONFIG_PATH` | `/etc/openfortivpn/config` | Path to openfortivpn config |
| `API_TCP_PORT` | `8080` | TCP port for API/Web UI |
| `API_UNIX_SOCKET` | `/run/openfortivpn/api.sock` | Unix socket path |
| `LOG_LEVEL` | `info` | `debug`, `info`, `warn`, `error` |

### Volume-Mounted Config

Users can mount a custom `openfortivpn.conf`:
```ini
host = vpn.avular.com
port = 443
saml-login = 8020
# trusted-cert = <hash>
```

The entrypoint merges env vars with the mounted config (env vars take precedence for `host`, `port`, `saml-login`).

## Signal Handling & Lifecycle

### Startup
1. `tini` starts `entrypoint.sh`.
2. `entrypoint.sh` validates config and environment.
3. Starts `dnsmasq` (listening on `127.0.0.1:53`).
4. Starts the Go control plane.
5. If `AUTO_CONNECT=true`, calls the internal connect routine.
6. Otherwise, waits in a `sleep infinity` loop, ready for API commands.

### Connect
1. API receives `POST /connect`.
2. Checks if `openfortivpn` is already running.
3. Generates `waybar.conf` (strips `saml-login` from config, adds it as CLI arg).
4. Spawns `openfortivpn --config /tmp/waybar.conf --saml-login=$SAML_PORT`.
5. Polls for port `$SAML_PORT` to be listening (max 10s).
6. Returns `saml_url` to caller.
7. Polls for `ppp0` to appear (max 30s).
8. On `ppp0` up: applies DNS (resolved or dnsmasq path).
9. State transitions to `connected`.

### Disconnect (API or Signal)
1. API receives `POST /disconnect` OR container receives SIGINT/SIGTERM.
2. Sends `SIGINT` to `openfortivpn` process.
3. Waits for `openfortivpn` and `pppd` to exit (max 10s).
4. If still running, escalates to `SIGTERM`, then `SIGKILL`.
5. Ensures `ppp0` is down (`ip link set ppp0 down`).
6. Restores host DNS (reverts `/etc/resolv.conf` or calls `resolvectl revert`).
7. State transitions to `disconnected`.

### Health Check
- `GET /health` returns 200 if the control plane is running.
- Docker/Podman health check calls this endpoint.
- If the VPN is expected to be connected but `ppp0` is down for >60s, health returns 503.

## Improvements Over Current Implementation

1. **No systemd dependency**: Works on Alpine, Void, Artix, and any non-systemd distro.
2. **Portable DNS**: Self-contained `dnsmasq` replaces `resolvectl` + `systemd-resolved` coupling.
3. **Clean API**: Replaces the 566-line monolithic bash script with a structured HTTP API.
4. **Signal safety**: Graceful shutdown via `tini` + explicit signal handlers replaces brute-force `pkill` chain.
5. **Health checks**: Container-native health checks detect and recover from stuck states.
6. **Multi-arch support**: Container image supports `amd64` and `arm64`.
7. **Configuration flexibility**: Env vars + volume mounts + API updates; no Ansible required.
8. **Better logging**: Centralized log streaming via API; no more `/tmp/openfortivpn.log` world-readable file.
9. **Cross-platform CLI**: `vpnctl` works the same on any machine with `curl`.
10. **Optional Web UI**: Provides a clickable interface without requiring Waybar/Hyprland.

## Known Limitations

1. **Linux-only for full-network VPN**: `--network host` is required for `ppp0` to be visible on the host. On macOS Docker Desktop and Windows WSL2, this does not expose the interface to the macOS/Windows host. A SOCKS5 proxy mode could be added as a future enhancement for macOS/Windows users.
2. **Privileged container required**: `pppd` needs `CAP_NET_ADMIN` and `/dev/ppp`. This is inherent to PPP VPNs and cannot be avoided.
3. **`/etc/resolv.conf` manipulation**: On hosts where `/etc/resolv.conf` is managed by `systemd-resolved` or `NetworkManager`, our direct manipulation may be overwritten by the host service. The `DNS_METHOD=resolved` path mitigates this, but requires `resolvectl` on the host.
4. **SAML port conflict**: If port `8020` is in use on the host, the SAML listener will fail. This is configurable via `SAML_PORT`.
5. **No automatic route conflict resolution**: If the VPN subnet overlaps with the host's local network, routing may need manual intervention.
